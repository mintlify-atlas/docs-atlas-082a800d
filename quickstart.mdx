---
title: "Quickstart: deploy your first tunasync mirror setup"
description: "Deploy a tunasync manager and worker, configure your first rsync mirror job, and control it with tunasynctl — up and running in under 10 minutes."
---

This guide walks you through the minimum configuration needed to run a working tunasync deployment: one manager, one worker, and a single mirror job syncing the [elvish](https://elv.sh) project via rsync. By the end you will have a running system you can query with `tunasynctl` and extend with additional mirrors.

<Steps>
  <Step title="Download or build tunasync">
    The simplest way to get started is to download a pre-built binary from the [GitHub releases page](https://github.com/tuna/tunasync/releases/latest). Releases include tarballs for Linux x86_64 and ARM64.

    If you prefer to build from source, you need Go 1.21 or later. Clone the repository and run:

    ```shell
    make all
    ```

    To build for a non-native architecture:

    ```shell
    make ARCH=linux-arm64 all
    ```

    Binaries are placed in `build-linux-amd64/` (or the equivalent directory for your target architecture). The build produces three binaries: `tunasync`, `tunasynctl`, and supporting files.
  </Step>

  <Step title="Create the manager config">
    The manager is a stateful HTTP server. Create the directory and config file:

    ```shell
    mkdir ~/tunasync_demo
    mkdir -p /tmp/tunasync
    ```

    Create `~/tunasync_demo/manager.conf`:

    ```toml
    debug = false

    [server]
    addr = "127.0.0.1"
    port = 12345
    ssl_cert = ""
    ssl_key = ""

    [files]
    db_type = "bolt"
    db_file = "/tmp/tunasync/manager.db"
    ca_cert = ""
    ```

    The manager stores all job state in the BoltDB file at `db_file`. The `ca_cert` field is only needed when workers connect over HTTPS. In this local setup both are left empty.

    <Note>
    In addition to `bolt`, tunasync supports `badger`, `leveldb`, and `redis` database backends. For Redis, set `db_type = "redis"` and set `db_file` to the connection URL: `redis://user:password@host:port/db_number`.
    </Note>
  </Step>

  <Step title="Create the worker config">
    The worker runs on the sync host and executes mirror jobs. Create `~/tunasync_demo/worker.conf`:

    ```toml
    [global]
    name = "test_worker"
    log_dir = "/tmp/tunasync/log/tunasync/{{.Name}}"
    mirror_dir = "/tmp/tunasync"
    concurrent = 10
    interval = 120

    [manager]
    api_base = "http://localhost:12345"
    ca_cert = ""

    [cgroup]
    enable = false
    base_path = "/sys/fs/cgroup"
    group = "tunasync"

    [server]
    hostname = "localhost"
    listen_addr = "127.0.0.1"
    listen_port = 6000
    ssl_cert = ""
    ssl_key = ""

    [[mirrors]]
    name = "elvish"
    provider = "rsync"
    upstream = "rsync://rsync.elv.sh/elvish/"
    use_ipv6 = false
    ```

    Key fields to understand:

    - `global.name` — the unique worker ID registered with the manager
    - `global.mirror_dir` — base directory where mirror data is written
    - `global.interval` — default sync interval in minutes (overridable per mirror)
    - `global.concurrent` — maximum number of jobs running at the same time
    - `manager.api_base` — the HTTP URL of the manager; must match the manager's `addr` and `port`
    - `server.listen_port` — the port this worker listens on for commands forwarded by the manager

    <Note>
    The `{{.Name}}` template in `log_dir` is expanded to each mirror's name at runtime, creating a separate log file per mirror.
    </Note>
  </Step>

  <Step title="Start the manager">
    Run the manager in a terminal or as a background process:

    ```shell
    tunasync manager --config ~/tunasync_demo/manager.conf
    ```

    The manager listens on `127.0.0.1:12345` and is ready to accept worker registrations. You can add `--verbose` for more log output or `--debug` to enable the Gin HTTP request log.
  </Step>

  <Step title="Start the worker">
    In a second terminal, start the worker:

    ```shell
    tunasync worker --config ~/tunasync_demo/worker.conf
    ```

    The worker registers itself with the manager at startup, then immediately begins scheduling the mirrors defined in its config. The elvish mirror will start syncing after a short delay.

    <Tip>
    Send `SIGHUP` to a running worker to reload its mirror configuration without restarting or interrupting any in-progress sync jobs:

    ```shell
    kill -HUP $(pidof tunasync)
    ```

    This lets you add or modify mirrors at runtime.
    </Tip>
  </Step>

  <Step title="Check status with tunasynctl">
    Query the manager for the current status of all jobs:

    ```shell
    tunasynctl list -p 12345 --all
    ```

    You will see a table showing each mirror, its worker, current state (`none`, `syncing`, `success`, etc.), last update time, and upstream URL.

    To avoid specifying `-p 12345` every time, create a `tunasynctl` config file at `~/.config/tunasync/ctl.conf` (or `/etc/tunasync/ctl.conf` for system-wide defaults):

    ```toml
    manager_addr = "127.0.0.1"
    manager_port = 12345
    ca_cert = ""
    ```

    With that in place you can run `tunasynctl list --all` without the port flag. The per-user file at `~/.config/tunasync/ctl.conf` takes precedence over the system file.
  </Step>
</Steps>

## HTTP vs HTTPS

<Note>
When the manager and all workers run on the same host (as in this guide), plain HTTP is sufficient and simpler to configure. Leave `ssl_cert` and `ssl_key` empty in both the manager and worker configs, and use `http://` as the scheme in `api_base`.

For production deployments where the worker communicates with the manager over a network, configure TLS: set `ssl_cert` and `ssl_key` in the manager config, set `ca_cert` in the worker config to the CA certificate, and change `api_base` to `https://`.
</Note>
