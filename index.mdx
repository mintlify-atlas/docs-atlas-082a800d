tunasync is the mirror job management system powering [TUNA mirrors](https://mirrors.tuna.tsinghua.edu.cn), one of the largest open-source software mirrors in China. It provides a manager-worker architecture where a central manager coordinates sync schedules and status across one or more worker processes, each running configurable mirror jobs using rsync, custom commands, or two-stage rsync providers.

## How it works

tunasync splits responsibilities between two components: the **manager** and the **worker**. The manager is a stateful HTTP server that tracks job status in a local database (BoltDB, Badger, LevelDB, or Redis). Workers register themselves on startup, then execute mirror jobs on a configurable schedule, reporting status back to the manager after each run.

<Steps>
  <Step title="Start the manager">
    Launch `tunasync manager` with a TOML config file. The manager exposes an HTTP API on port 14242 by default and stores job state in a BoltDB file.
  </Step>
  <Step title="Start one or more workers">
    Launch `tunasync worker` on each sync host. Workers register with the manager, load their mirror configs, and begin scheduling jobs immediately.
  </Step>
  <Step title="Control jobs with tunasynctl">
    Use `tunasynctl` to list job status, start or stop individual mirrors, reload configs, and manage workers — all without restarting any daemon.
  </Step>
</Steps>

## Key features

<CardGroup cols={2}>
  <Card title="Multiple sync providers" icon="layer-group">
    rsync, two-stage rsync (for Debian-style repos), and arbitrary shell commands. Mix providers across mirrors in the same worker.
  </Card>
  <Card title="Resource isolation" icon="gauge">
    Limit memory per mirror job using cgroup v1 or v2, or run jobs inside Docker containers for full isolation.
  </Card>
  <Card title="Atomic snapshots" icon="camera">
    Btrfs and ZFS snapshot hooks create atomic mirror views so users never see partially-synced data.
  </Card>
  <Card title="Hot config reload" icon="rotate">
    Send SIGHUP to a worker to reload mirror configurations without interrupting in-progress sync jobs.
  </Card>
</CardGroup>
